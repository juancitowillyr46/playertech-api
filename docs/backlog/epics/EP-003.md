# EP-003 Gestión de Usuarios y Accesos

## Objetivo
Controlar el ciclo de vida de usuarios y accesos en PlayerTech para plataforma y tenant, sin mezclar responsabilidades entre `ROLE_ROOT` y usuarios de academia.

## Problema que Resuelve
Garantizar que únicamente usuarios autorizados accedan al sistema, con reglas claras para identidad, pertenencia a academia y administración de permisos.

## Valor de Negocio
Protección de la información, gobierno seguro de identidades y administración consistente del acceso multi-tenant.

## Alcance

### Plataforma
* Crear usuarios root.
* Listar usuarios de plataforma.
* Actualizar usuarios de plataforma.
* Activar y desactivar usuarios de plataforma.

### Tenant
* Crear usuarios administrativos del tenant por invitación.
* Enviar y reenviar invitaciones por correo.
* Activar cuenta y definir contraseña.
* Consultar el perfil del usuario autenticado.
* Actualizar solo el nombre del usuario autenticado.
* Solicitar y confirmar restablecimiento de contraseña.
* Listar usuarios del tenant actual.
* Actualizar usuarios del tenant actual.
* Activar y desactivar usuarios del tenant actual.
* Respetar el contexto `academy_id`.

## Reglas de Negocio Relacionadas
* Solo usuarios activos pueden autenticarse.
* Cada usuario posee un rol.
* Los usuarios tenant pertenecen a una academia.
* Los usuarios root no pertenecen a ninguna academia.
* `ROLE_ACADEMIC_ADMIN` es el único rol funcional para gestionar usuarios administrativos de academia.
* Un tenant debe conservar al menos un usuario administrador activo.
* Suspender una academia bloquea el acceso de todos sus usuarios.
* Desactivar un usuario no desactiva la academia.
* No se puede desactivar el último administrador activo del tenant.
* El usuario owner/admin inicial del tenant se crea como parte del onboarding o del alta desde plataforma.
* Un usuario root nunca debe ser convertido en usuario tenant mediante este flujo.

## Flujo de Negocio
1. Crear usuario según el contexto.
2. Asignar rol.
3. Asociar `academy_id` cuando el usuario sea tenant.
4. Enviar invitación por correo cuando el alta sea para un usuario administrativo de academia.
5. Usuario define contraseña y confirma su cuenta.
6. Usuario inicia sesión.
7. Usuario accede según permisos.
8. Usuario se activa, desactiva o actualiza según su contexto.

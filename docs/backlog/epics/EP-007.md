# EP-007 Gestión de Jugadores

## Objetivo

Administrar el ciclo base del jugador dentro de una academia.

## Problema que Resuelve

Centraliza la información principal del jugador para que el dominio pueda crecer sobre una base consistente.

## Valor de Negocio

El jugador es la entidad central del dominio funcional de PlayerTech.

## Actores

* Academic Administrator
* Super Admin

## Dominios Involucrados

* Player
* Academy

## Reglas de Negocio Relacionadas

* Todo jugador pertenece a una academia.
* El jugador debe quedar aislado por `academy_id`.
* Un jugador puede tener múltiples acudientes.
* Un jugador puede tener múltiples matrículas históricas.
* La desactivación del jugador debe ser lógica, no física.
* El contexto tenant no debe poder ver ni modificar jugadores de otra academia.

## Flujo de Negocio

1. Registrar jugador dentro de la academia.
2. Consultar jugadores del tenant.
3. Consultar detalle de un jugador.
4. Actualizar datos del jugador.
5. Gestionar estado del jugador: desactivar y reactivar.
6. Importar jugadores y categorías desde Excel.
7. Resolver categorias mediante `category_key` estable para importaciones e integraciones.

## Historias Candidatas

* Registrar jugador.
* Consultar jugador.
* Listar jugadores.
* Actualizar jugador.
* Gestionar estado del jugador.
* Importar jugadores en lote.
* Definir clave de negocio estable para categorias.

## Base Técnica de Referencia

* `PlayerId` debe seguir la convención de custom type UUID.
* `Player` debe mapearse con XML puro.
* Los DTOs de entrada deben validarse formalmente.
* Los handlers deben permanecer delgados.
* Las respuestas deben devolver DTOs, no entidades Doctrine.
* El borrado físico no aplica para el dominio de negocio.

## Métricas de Éxito

* Jugadores registrados.
* Jugadores activos por academia.
* Cero fugas cross-tenant.

## Fuera de Alcance

* Relaciones jugador-acudiente.
* Matrículas.
* Asignaciones a equipos.
* Pagos.
